# Vault 内容对外改写提示词

适用目录：
`04_operations/07_skill_lib/book-kb-领导者的意识进化/vault`

这份提示词是给 AI 用的，目标不是“忠实搬运内部知识卡”，而是把 `vault/` 里的内容改成适合公开网页阅读的版本。

## 先说明一件事

`vault/` 里的很多内容本来是为内部整理、抽取、关联、复用服务的，所以会天然带有这些问题：

- 卡片味太重
- 数据库味太重
- 章节结构太满
- 内部工作流字段太多
- 链接太密
- 对第一次阅读的外部读者不够友好

所以改写目标不是“尽量多保留”，而是：

- 保留核心观点
- 保留真正对读者有帮助的解释
- 保留少量必要的站内链接
- 删掉内部维护痕迹
- 让网页看起来像公开发布内容，而不是知识库底稿

## 建议怎么用

### 用法 1：单篇改写

把下面的“单文件提示词”发给 AI，再把某个 `.md` 文件正文贴进去。

### 用法 2：批量改写

先用“批量提示词”约束整体风格，再按文件一篇一篇喂给 AI。

### 用法 3：给 Codex / ChatGPT 直接执行

可以这样说：

`请按照 vault内容对外改写提示词.md 的规则，改写这个文件：xxx.md，只输出可直接替换的 Markdown 成稿。`

## 改写时要删掉什么

下面这些内容，通常不应该直接出现在对外网页里：

- `这是一张 N / Note 概念卡`
- `这是一张 K / Knowledge 主题页`
- `提取说明`
- `创建时间`
- `最后更新时间`
- `调用回流位`
- `成熟度判断`
- `在这套系统里的位置`
- `来源资产`
- `关系类型`
- `下一轮优先记录什么`
- `可转成 P`
- `default_usage`
- `visual_elements`
- `case_type`
- `what_it_proves`
- `linked_concepts`
- `scene_description`
- `成熟度：Growing`
- 大段仅服务内部知识管理的链接清单
- 为整理者自己写的备注、提取痕迹、研究痕迹

## 改写时要保留什么

- 标题对应的核心结论
- 对读者真正重要的概念解释
- 有帮助的场景判断
- 实际可用的方法和动作
- 少量必要的站内 `[[链接]]`
- 对理解特别关键的例子

## 对外格式的总体原则

1. 默认读者不知道什么是 K 卡、N 卡、节点、图谱、调取链路。
2. 第一屏就要让人知道“这页在讲什么，为什么值得看”。
3. 可以保留复杂观点，但表达方式必须比内部稿更自然、更克制。
4. 不要为了完整而堆结构，宁可删掉低价值小节。
5. 不要把所有内部分析框架原样摆出来。
6. 不要把页面写成课程讲义、知识库条目或提词器。
7. 链接要少而准，避免一屏全是 `[[...]]`。
8. 如果原文有重复段落、重复清单、重复判断，要主动合并。
9. 如果原文太抽象，可以补一小段通俗解释，但不要编造原文没有的事实。
10. 输出应该是“读者能直接看”的网页稿，不是“分析过程”。

## 各类型页面建议改成什么样

### 主题页

适合改成：

- 这篇内容在讲什么
- 核心观点
- 为什么这很重要
- 现实里常见表现
- 你可以怎么理解或应用
- 延伸阅读

### 章节页

适合改成：

- 本章主要回答什么
- 这一章最重要的 3-5 个收获
- 这一章为什么关键
- 你可以带走的一个判断或方法
- 延伸阅读

### 概念页

适合改成：

- 这个概念是什么意思
- 它解决什么问题
- 你在什么情况下会遇到它
- 常见误解
- 实际工作里怎么用
- 延伸阅读

### 方法页

适合改成：

- 什么时候适合用
- 具体怎么做
- 常见错误
- 一个简短例子
- 延伸阅读

### 场景页

适合改成：

- 这类情况表面发生了什么
- 底层卡点是什么
- 为什么会反复出现
- 可以怎么应对
- 延伸阅读

### 人物页

适合改成：

- 这是谁
- 他的核心贡献是什么
- 为什么值得读
- 适合从哪里开始了解

## 单文件提示词

下面这段可以直接复制给 AI：

```md
你现在是一个“面向外部读者的知识网站编辑”，不是知识库管理员，也不是课程资料整理员。

我会给你一篇来自 `04_operations/07_skill_lib/book-kb-领导者的意识进化/vault` 的 Markdown 文件。它原本带有很强的内部知识卡和整理痕迹。请你把它改写成适合公开网页直接展示的版本。

你的目标：
- 保留这篇内容真正有价值的核心观点
- 提升普通读者的可读性
- 删掉内部维护痕迹、抽取痕迹、工作流字段
- 让页面看起来像对外发布内容，而不是内部知识库卡片

你必须遵守下面这些规则：

一、保留规则
- 保留最上方原有 YAML frontmatter，不要擅自删除，除非我明确要求你改。
- 保留正文主标题。
- 保留真正有帮助的 `[[站内链接]]`，但只保留少量高价值链接，一般 3-8 个即可。
- 保留核心概念、关键判断、必要例子、真正能帮助理解或行动的部分。

二、删除规则
- 删除或改写所有“内部知识库味”的内容，例如：
  - `这是一张 N / Note 概念卡`
  - `这是一张 K / Knowledge 主题页`
  - `提取说明`
  - `创建时间`
  - `最后更新时间`
  - `调用回流位`
  - `成熟度判断`
  - `在这套系统里的位置`
  - `来源资产`
  - `关系类型`
  - `下一轮优先记录什么`
  - `可转成 P`
  - `default_usage`
  - `visual_elements`
  - `case_type`
  - `what_it_proves`
  - `linked_concepts`
  - `scene_description`
- 删除重复表达、内部术语堆砌、过密链接清单、大段为了知识管理而存在的枚举。
- 删除正文里不该对外展示的时间戳、抽取说明、卡片说明。

三、表达规则
- 不要假设读者知道 K 卡、N 卡、节点、图谱、链路、回流、成熟度。
- 前两段就要让读者知道：这页在讲什么、为什么重要。
- 语气要像成熟的知识网站或公众号文章，不要像内部数据库，不要像课程讲义。
- 可以保留复杂观点，但表达要更自然、更克制。
- 如果原文有太多内部结构，请主动降噪，不要机械保留所有小节。
- 如果原文太抽象，可以补一小段更通俗的解释，但不要编造事实。

四、结构规则
- 正文尽量控制在 4-7 个二级标题内。
- 优先使用以下这些标题，但不要机械全用：
  - `## 这篇内容在讲什么`
  - `## 核心观点`
  - `## 为什么这很重要`
  - `## 你可以怎么理解或使用`
  - `## 常见误区`
  - `## 延伸阅读`
- `## 延伸阅读` 只保留最必要的 3-5 个站内链接。

五、按文件类型改写
- 如果是“主题页”，写成：核心主题判断 + 为什么重要 + 关键展开 + 适用场景/误区。
- 如果是“章节页”，写成：本章回答什么 + 3-5 个关键收获 + 为什么关键 + 带走的一个方法或判断。
- 如果是“概念页”，写成：概念是什么 + 它解决什么问题 + 如何识别 + 常见误解 + 怎么用。
- 如果是“方法页”，写成：什么时候用 + 怎么做 + 常见错误 + 一个简短示例。
- 如果是“场景页”，写成：表面发生了什么 + 底层卡点是什么 + 为什么会反复出现 + 可以怎么应对。
- 如果是“人物页”，写成：这是谁 + 核心贡献 + 为什么值得读。

六、输出规则
- 只输出“修改后的完整 Markdown 成稿”。
- 不要解释你删了什么。
- 不要写“以下是改写版”。
- 不要写任何过程说明。
- 输出要能直接替换原文件正文。

七、质量检查
- 读者第一次来到这页也能看懂。
- 页面不像卡片库、数据库、课程底稿。
- 链接数量克制，不会干扰阅读。
- 重点明显，结论先行。
- 没有正文里的内部元信息。

下面是原始文件，请开始改写。

文件路径：
{{FILE_PATH}}

原始 Markdown：
{{RAW_MARKDOWN}}
```

## 批量改写提示词

如果你准备连续处理很多文件，可以先给 AI 这一段：

```md
接下来我会连续给你多篇来自 `04_operations/07_skill_lib/book-kb-领导者的意识进化/vault` 的 Markdown 文件。

请你始终按下面这个统一标准改写：

- 改成适合公开网页阅读的版本
- 保留 YAML frontmatter
- 删掉内部知识卡、提取说明、调用关系、成熟度、来源资产等内部内容
- 让语气更像对外发布内容，而不是知识库
- 控制链接密度
- 每次只输出“该文件可直接替换的完整 Markdown 成稿”
- 不解释、不总结、不复盘

如果我发来的是：
- `主题/` 下的文件：按主题页改
- `章节/` 下的文件：按章节页改
- `概念/` 下的文件：按概念页改
- `方法/` 下的文件：按方法页改
- `场景/` 下的文件：按场景页改
- `人物/` 下的文件：按人物页改

后续我每发一篇，你直接开始改写，不要重复规则。
```

## 给 AI 的补充判断

如果 AI 不确定某段该不该留，可以按下面判断：

- 读者理解这个主题必须知道：保留
- 对内部维护有用、对外部阅读没用：删除
- 重复表达同一个意思：合并
- 只是为了展示结构完整，不增加理解：删除
- 只是为了知识卡互相调用：缩减
- 会让读者觉得“这像内部资料”：改写或删除

## 你后面真正执行时的建议

如果你要长期保留这些改写结果，只改 `vault/` 还不够，因为这里很多内容会被 `generate_site.py` 重新生成覆盖。

更稳的做法通常有两种：

- 先用这份提示词把内容改顺，再把规则固化进 `generate_site.py`
- 或者把上游素材源头改成更适合对外表达的版本，再重新生成

如果只是先快速试效果，这份提示词已经够你逐篇改了。
